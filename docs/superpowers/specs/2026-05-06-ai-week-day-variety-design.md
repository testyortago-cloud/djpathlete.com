# AI Week/Day Generation — Variety, Difficulty, and Technique Improvements — Design

**Date:** 2026-05-06
**Status:** Approved design; awaiting implementation plan
**Author:** Darren Paul (with Claude Code)
**Builds on:** [2026-04-14 AI generation improvements](2026-04-14-ai-generation-improvements-design.md)

## Problem

The week-by-week and day-by-day AI generators (`functions/src/ai/week-orchestrator.ts`) produce programs that need substantial manual correction. Vikrim — a real client — keeps receiving the same or near-identical exercises across:

1. Consecutive weeks of his current program (worst pain).
2. New programs generated for him over time.
3. Other clients with similar profiles, generated by the same coach.

Net effect: the AI feature is currently a *time sink*, not a time saver. The coach spends more time fixing the output than designing from scratch.

The 2026-04-14 spec addressed these problems for the full-program generator (`generateProgramSync`) but the week/day path was never brought to parity. Two newer levers (`coach_ai_policy`, `generated_exercise_usage`) exist in the database and are read by the full-program path, but the week/day path ignores both.

## Root Causes (week/day path)

| # | Issue | Root cause | Location |
|---|---|---|---|
| 1 | Cross-program memory loss | Week orchestrator never reads `getCoachRecentUsageFromFn` or `getClientRecentUsageFromFn`, and never calls `recordUsageFromFn` after saving. The system has zero memory across program/week regenerations. | `functions/src/ai/week-orchestrator.ts:295-345, 718-754` |
| 2 | AVOID list is prose, not data | The candidate library JSON shown to Agent 3 still contains every exercise the prompt says to avoid. Models drift back to high-relevance "forbidden" picks. | `functions/src/ai/week-orchestrator.ts:660-665`; `dedup-verify.ts:115-160` |
| 3 | No real Agent 1 in week mode | Mock analysis hard-codes `technique_plan: ["straight_set"]` and a fixed difficulty ceiling. Phase progression and technique variation logic never run on week/day generation. | `functions/src/ai/week-orchestrator.ts:551-585` |
| 4 | Crude slot-role inference | `buildPriorContextFromExistingExercises` infers role from `order_index` (0 = warm_up, 1–2 = compound, else accessory). Real role/movement_pattern/target_muscles get squashed into wrong groups, weakening the AVOID-list signal. | `functions/src/ai/week-orchestrator.ts:617-633`; `dedup-verify.ts:78-110` |
| 5 | Same top scorers always win | `semanticFilterExercises` and `scoreAndFilterExercises` use deterministic scoring with no diversity term. The same ~80 of 900+ exercises rise to the top every regeneration. | `functions/src/ai/exercise-filter.ts:290-451` |
| 6 | Coach AI Policy ignored | Week orchestrator never calls `getCoachPolicyFromFn` — coach-wide preferences (e.g., "no supersets", preferred techniques) are silently discarded. | `functions/src/ai/week-orchestrator.ts` (absent) |

## Design Decisions

| # | Decision | Rationale |
|---|---|---|
| 1 | **Bring week/day path to parity with full-program path.** Wire `coach_ai_policy`, `coachUsage(60d)`, `clientUsage(90d)` reads at the start; wire `recordUsageFromFn` write at the end. | The plumbing already exists for the full-program path. We're not building anything new — we're using what's there. |
| 2 | **Hard-prune already-used exercise IDs from the candidate library.** Already-used exercises are *removed* from the JSON shown to Agent 3, not just labeled "AVOID". | Eliminates model drift entirely on the no-repeat constraint. Defense in depth: the AVOID prose stays as a backup signal. |
| 3 | **Promote the week-mode mock analysis to a real Agent 1 call.** New `WEEK_PROFILE_ANALYZER_PROMPT` produces single-week `technique_plan` and `difficulty_ceiling`, scoped to the new week and informed by program history + coach policy. | Phase progression (e.g., week 5 introduces antagonist supersets, week 8 adds rest_pause) actually happens. +5–10s per generation, but this is the cost that converts the AI from net-negative to net-positive time. |
| 4 | **Add `slot_role` column to `program_exercises`.** Stop inferring role from `order_index`. Backfill existing rows. | Restores the dedup signal. Once real role data exists, AVOID list groups exercises correctly. |
| 5 | **MMR diversification in `semanticFilterExercises`.** Top-N candidates are picked by maximal marginal relevance (relevance × diversity), not pure relevance. | Library utilization grows organically as the 900+ pool expands. New exercises actually get picked instead of always losing to the same top scorers. |
| 6 | **Both algorithmic upgrades (hard-prune, MMR) ship via the shared filter functions.** | Full-program path inherits the improvements at zero additional implementation cost. |

## Architecture

Same 4-agent main pipeline, with the week/day pipeline upgraded from 2 agents to 3 agents:

**Before:** `Architect (Agent 2) → Selector (Agent 3)` with mock analysis.
**After:** `Profile Analyzer (Agent 1, week-scoped) → Architect (Agent 2) → Selector (Agent 3)`.

Both pipelines share four reinforcement layers:
1. **Prompt layer** — Agent 1 produces explicit `technique_plan` + `difficulty_ceiling`; Agent 2 must follow them; Agent 3 selects within the ceiling.
2. **Schema layer** — Existing Zod schemas reject violations and trigger retries (max 2).
3. **Data layer** — `generated_exercise_usage` records every assigned exercise; `coach_ai_policy` holds per-coach preferences. Both already exist.
4. **Selection layer** — `semanticFilterExercises` reads usage history, hard-prunes excluded IDs, MMR-diversifies the top-N.

```
BEFORE (week/day mode)                  AFTER (week/day mode)
─────────────────────────────────       ──────────────────────────────────────
fetch program + exercises               fetch program + exercises + coach_policy + usage_history
fetch profile                           fetch profile
                                        ┌── NEW: Agent 1 (week-scoped) ──┐
                                        │   inputs: profile + policy +   │
                                        │           program ctx + week#  │

                                        └────────────────────────────────┘
Agent 2 (architect)                     Agent 2 (architect, constrained by Agent 1 + policy)

```

## Component Changes

| Area | File | Change |
|---|---|---|
| Week orchestrator | `functions/src/ai/week-orchestrator.ts` | Replace mock-analysis block with Agent 1 call. Fetch `coach_ai_policy` + `coachUsage(60d)` + `clientUsage(90d)` upfront in parallel. Pass usage maps + `excludeIds` into `semanticFilterExercises`. Persist `slot_role` on insert. After successful save, fire-and-forget call to `recordUsageFromFn`. |
| Week-mode Agent 1 prompt | `functions/src/ai/prompts.ts` | New `WEEK_PROFILE_ANALYZER_PROMPT`. Smaller than the full-program version — only outputs single-week `session_structure`, `technique_plan`, `difficulty_ceiling`, and progression notes. Reuses the existing JSON shape so existing schema validation works unchanged. |
| Hard candidate pruning | `functions/src/ai/exercise-filter.ts` | `scoreAndFilterExercises` and `semanticFilterExercises` accept optional `excludeIds: Set<string>`. Filter applied after scoring/matching, before `ensurePatternBalance` (so patterns can still backfill from the unused pool). |
| MMR diversification | `functions/src/ai/exercise-filter.ts` | New helper `diversifyByMMR(scored, k, lambda)`. Similarity over `(movement_pattern, primary_muscles[], equipment_required[], training_intent[])`. Default `lambda=0.7`. Wired into both `semanticFilterExercises` and `scoreAndFilterExercises`. |
| Slot-role storage | `supabase/migrations/<ts>_program_exercise_slot_role.sql` | Add `slot_role text` to `program_exercises`. Backfill existing rows from current order_index inference. Idempotent migration. |
| Slot-role write path | `functions/src/ai/shared-helpers.ts` | `buildExerciseRows` writes `slot_role` from `slotDetailsLookup` so new generations populate the column. |
| Slot-group inference | `functions/src/ai/dedup-verify.ts` | `buildPriorContextFromExistingExercises` reads `slot_role` when present, falls back to current inference for legacy rows. Returns AVOID list as both prose AND a `Set<string>` for hard pruning. |
| Exclude-ID helper | `functions/src/ai/shared-helpers.ts` | New `buildExcludeIdSet(priorContext, currentSlotRoles)`. Returns IDs to physically remove from the candidate library, scoped to working slot roles being generated (warm_up/cool_down anchors are never excluded). |
| Tests | `functions/src/ai/__tests__/week-orchestrator.test.ts` (new), `__tests__/ai-filter-mmr.test.ts` (new), extend `exercise-filter.test.ts` and `exercise-context.test.ts` | Cover usage-history wiring, hard-prune behavior, MMR distribution, week-mode Agent 1 output, slot_role persistence and dedup. |

### Files NOT touched

- `orchestrator.ts` (full-program) — gets the algorithmic upgrades for free via shared filter functions.
- `validate.ts`, `schemas.ts`, `anthropic.ts`, `embeddings.ts`, `rag.ts`, `usage-history.ts`, `coach-policy.ts` — already correct.
- Admin UI — no new surfaces; the Coach AI Policy admin page from 2026-04-14 already exists.

### Scope guardrails

- No new tables. Reuses `generated_exercise_usage` and `coach_ai_policy` from 2026-04-14.
- No new external dependencies. MMR is ~30 lines of TypeScript over existing data.
- Backward compatible. All new filter parameters are optional with sensible defaults.

## Data Flow (one week/day generation)

```
Admin triggers week/day generation (app/api/admin/programs/[id]/generate-week/route.ts)
  ↓
Firebase job handler → generateWeekSync(request, requestedBy, jobId)
  ↓
1. Parallel fetch:
   - getProgramById, getProgramExercises, getExercisesForAI
   - getClientProfile (if client_id and !ignore_profile)
   - getCoachPolicyFromFn(requestedBy)            [NEW]
   - getCoachRecentUsageFromFn(requestedBy, 60)   [NEW]
   - getClientRecentUsageFromFn(client_id, 90)    [NEW]
  ↓
2. Build prior context + slot-role-aware AVOID set:
   priorContext = buildPriorContextFromExistingExercises(...)  // reads slot_role
   excludeIds   = buildExcludeIdSet(priorContext, targetSlotRoles)
  ↓
3. NEW Agent 1 (week-scoped Profile Analyzer):
   Input:  profile + program summary + coach policy + coach instructions + target week #
   Output: { session_structure, technique_plan[1], difficulty_ceiling[1], notes }
   Schema-validated; up to 2 retries; on 3rd failure fall back to mockAnalysis
  ↓
4. Agent 2 (week architect):
   Input: program context + Agent 1 output + coach policy + coach instructions
   Constraint: every slot.technique ∈ technique_plan[0].allowed_techniques
   Output: 1-week skeleton with slot_role on every slot
  ↓
5. Filter pipeline:
   filterByDifficultyLevel → filterByProgressionPhase → filterByInjuredJoints
   semanticFilterExercises({ coachUsage, clientUsage, excludeIds, mmrLambda: 0.7 })
     ├── semantic match against slot embeddings
     ├── apply usage penalties (coach -30, client -50, diversity +10)
     ├── REMOVE excludeIds entirely (hard prune)
     ├── MMR-diversified top-N
     └── ensurePatternBalance backfill from non-excluded library
  ↓
6. Agent 3 (selector — existing prompt):
   Input: skeleton + filtered library (no excluded IDs) + AVOID prose (defense in depth)
   Up to 2 retries on dedup or schema failure
  ↓
7. validateProgram (existing): joint load, equipment, ceiling, plane balance
  ↓
8. Save program_exercises (writes slot_role)
  ↓
9. fire-and-forget: recordUsageFromFn({ coach_id, client_id, program_id, rows })
```

### Key invariants

- `excludeIds` never includes warm_up or cool_down exercise IDs (matches existing `ANCHOR_ROLES` policy).
- `excludeIds` is computed per slot-role group — a "primary_compound | squat | quads" exclusion does not leak into "isolation | rotation | obliques".
- Coach policy flows through both Agent 1 and Agent 2 — Agent 1 honors it for technique_plan selection, Agent 2 honors it for skeleton structure.
- MMR runs after usage penalties — diversity boosts compound on top of usage penalties.

## Error Handling

| Failure | Behavior | Why |
|---|---|---|
| Coach policy fetch fails | Continue with empty policy, log warning | Tracking-layer failures must never block generation |
| Usage history fetch fails | Continue with empty maps, log warning | Degrades gracefully to current behavior |
| Agent 1 (week-mode) schema rejection | Retry up to 2x with feedback; on 3rd failure fall back to mockAnalysis | Preserves "always returns something" contract |
| Agent 2 schema rejection (technique violation) | Retry up to 2x with feedback; on 3rd failure surface as job warning | Existing behavior |
| Library gap after exclude pruning (zero candidates for a slot) | Re-run filter for that slot only, without `excludeIds`; warn in `substitution_notes` | Repeating one exercise beats failing the week |
| MMR returns < `MIN_EXERCISES` | Skip MMR, fall back to score-sorted top-N | MMR is an optimization, not a correctness requirement |
| `recordUsageFromFn` fails after save | Log warning; do not retry; do not block response | Generated program is already saved |
| Backfill of `slot_role` for legacy rows fails | Migration is idempotent and resumable; log per-row errors | Backfill must not be a launch blocker |

## Testing Strategy

| Level | Test | Asserts |
|---|---|---|
| Unit | `buildExcludeIdSet` with 3 prior weeks of accessory rows | Returns IDs for matching slot-role groups; warm_up/cool_down absent |
| Unit | `diversifyByMMR` with 200 candidates, lambda=0.7, k=80 | Result ≤ 80; distinct movement_patterns ≥ 6; no duplicates |
| Unit | `semanticFilterExercises({ excludeIds })` | Excluded IDs absent from output even with highest semantic score |
| Unit | `filterByProgressionPhase` (regression) | Beginner week 1: zero intermediates; week 3+: low-score intermediates included |
| Unit | `buildPriorContextFromExistingExercises` reads `slot_role` when present, falls back to inference when null | Both branches covered |
| Integration | `generateWeekSync` for week 5 of an existing 4-week program | < 3% repetition score; ≥ 80% of working exercises new; usage rows recorded |
| Integration | `generateWeekSync` with `coach_ai_policy.disallowed_techniques: ["superset"]` | Zero supersets in skeleton OR exercise notes |
| Integration | `generateWeekSync` for a beginner client | All exercises difficulty ≤ beginner OR (intermediate ∧ score ≤ 4) when week ≥ 3 |
| Integration | Generate 4 weeks for client A, then 4 weeks for client B (similar profiles, same coach) | < 50% exercise overlap between clients |
| E2E (Playwright) | Coach generates week 5 in admin UI | UI shows ≥ 80% novel exercises in week 5 view |

## Parallel Execution Plan

**Wave 1 (4 independent agents, all parallel):**

- Agent A — Week-mode Agent 1 prompt in `prompts.ts` (`WEEK_PROFILE_ANALYZER_PROMPT`)
- Agent B — Hard-prune + MMR in `exercise-filter.ts` (new optional params; `diversifyByMMR` helper)
- Agent C — Slot-role migration + backfill in `supabase/migrations/`
- Agent D — Slot-role-aware dedup updates in `dedup-verify.ts`

**Wave 2 (2 agents, after Wave 1):**

- Agent E — `buildExcludeIdSet` + `slot_role` write path in `shared-helpers.ts` (depends on C, D)
- Agent F — Week orchestrator wiring in `week-orchestrator.ts` (depends on A, B, D, E)

**Wave 3 (1 agent, after Wave 2):**

- Agent G — Tests (unit + integration + e2e; depends on everything else)

## Out of Scope

- Changes to the full-program orchestrator beyond inheriting the algorithmic upgrades.
- New tables or columns beyond `program_exercises.slot_role`.
- Model changes (no fine-tuning, no provider switch, no prompt-cache restructure).
- Exercise library changes (no edits to the 900+ exercise records).
- Admin UI changes — Coach AI Policy admin page already exists.
- Multi-tenant/franchise coach isolation beyond a single `coach_id` scope.
- Re-validating outputs of the existing `validate.ts` agent — its checks already cover what we need.
